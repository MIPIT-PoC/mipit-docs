### Dimensión 3 — Code quality + architecture (A3)

**Top 5 críticos:**
1. SSE PII sin auth (C1 consolidado)
2. Double-registered RMQ consumers (C9)
3. `reconciliation.findSince` carga toda tabla en memoria (sin LIMIT)
4. DLQ requeue infinito (`dlq-handler.ts:84` `nack(false, true)` sin retry counter)
5. Race en idempotency 2nd lookup → respuesta sintética `{status: 'QUEUED'}` hardcoded

**Architecture mayor (pre-producción):**
- R1. `@mipit/adapter-runtime` package compartido (recortar 300 LOC duplicada en 3 workers)
- R2. `@mipit/contracts` versionado (Zod single SoT → OpenAPI + JSON Schema + TS types)
- R3. Idempotency middleware (no en route handler)
- R4. Outbox pattern para webhooks (garantía at-least-once con tabla `webhook_outbox`)
- R5. Circuit breaker integrado en publisher + adapter HTTP clients (hoy solo en `ui-proxy`)
- R6. Property-based tests con fast-check (invariantes: roundtrip translation, FX symmetry)
- R7. DI container (eliminar singletons `db.ts:4`, `health.ts:21`, `sse.ts:26`, `circuit-breaker.ts:203`)
- R8. OpenTelemetry alineado vía pnpm workspaces (drift core ↔ adapters de 6+ meses)

## Veredicto final

### ¿La tesis cumple lo prometido?

**Sí, con caveats documentados.** 19/20 RF cumplidos. La interoperabilidad técnica está demostrada con smoke 3/3 rail-pairs COMPLETED.

### ¿Es defendible ante panel técnico?

**Sí**, con dos condiciones:
1. **Aplicar Bloque A (~3h hardening)** antes de la defensa → cierra 14 issues visibles
2. **Anticipar las preguntas Q4, Q9, Q21, Q27, Q29** con respuestas honestas: "mapping_table es documentación, compensación es declarativa para PoC, alerts→core flow requires `webhooks/alertmanager` que estamos completando, Jaeger link usa attribute search post-fix, histogram buckets se ajustan"

### Aspectos donde MiPIT brilla (mostrar primero en demo)

- Pipeline 8-pasos con audit trail granular
- Idempotency criptográfica (SHA256 sobre payload, 409 con cuerpo distinto)
- UETR + ChrgBr + IntrBkSttlmDt **persistidos** (post-H1 también visibles en UI)
- Smoke 3/3 rail-pairs COMPLETED end-to-end
- Generators con checksums **reales** (CPF mod-11, CLABE mod-10, NIT mod-11)
- PII redaction en logs (Pino redact configurado correctamente)
- Publisher confirms wired (RabbitMQ ConfirmChannel)
- Bre-B como **case study de extensibilidad arquitectónica** (agregado completo en P04)
- LIMITATIONS.md como **honestidad académica explícita**

### Aspectos donde MiPIT requiere narrativa cuidadosa

- `mapping_table` SQL es documentación, no engine (anticípalo, no esperes que lo descubran)
- Compensación es declarativa (pacs.004 stub); aceptable para PoC si se declara
- 3 mocks (PIX/SPEI/BREB) son simulación; PIX y BREB con endpoints inventados sobre specs reales; SPEI con OAuth ficcional
- FX usa fallback rates hardcoded sin API key; pre-demo enunciar "PoC usa rate fallback; producción usaría Open Exchange Rates"
- Tests "11/11 verde" cuenta 3 historical entries + tests "live" que skipean en CI → no es 11 corridas reales

### Roadmap post-tesis (si MiPIT continúa)

Aplicar Bloque C completo (5-10 días) + Bloque B (3.5 días) llevaría a MiPIT a un **MVP defendible** ante revisores ISO 20022 / SWIFT CBPR+. El gap principal restante sería **conexión a sandboxes reales** (BCB DICT, Banxico STP, BanRep BanRep TR-002 directorio), lo cual requiere licencias financieras fuera del scope académico — pero la arquitectura ya está lista para conectarlas.
